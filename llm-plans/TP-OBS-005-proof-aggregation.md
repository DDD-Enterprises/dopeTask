# TP-OBS-005: Proof Aggregation & Audit Surface

## Objective
Aggregate the deterministic execution proofs generated by the `GeminiExecutor` (TP-CORE-001) into a fully compliant, standardized audit surface. Ensure the generated `_PROOF_BUNDLE.json` and `_PROOF_ARCHIVE.zip` strictly follow the canonical schemas defined in `docs/proof/`.

## Background & Motivation
The core execution engine currently writes raw proof files (`proof/{tp_id}_PROOF.json`), but these do not yet conform to the comprehensive top-level schema (`DOPETASK_BUNDLE_SCHEMA.md`) required for enterprise observability, CRM user stories, and human supervisor reviews. We need a dedicated aggregation layer to finalize and bundle these artifacts.

## Scope & Impact
- Creates or updates `src/dopetask/obs/proof_aggregator.py`.
- Integrates with the execution lifecycle to translate `TPProofBundle` output into the strict `Dopetask Proof Bundle Schema`.
- Automatically generates the `_PROOF_ARCHIVE.zip` containing all underlying code changes, logs, and artifacts, alongside a `PROOF_ARCHIVE_MANIFEST.json`.

## Proposed Solution

### 1. Schema Alignment (`proof_aggregator.py`)
Implement an aggregator that reads the raw execution outputs and constructs the final bundle:
- `tp_id`, `status` (VALIDATED, FAILED).
- `summary`: Automatic summarization of `result` and `key_findings` based on step success rates.
- `acceptance_checks`: Extract passed/failed checks from the executed step validations.
- `validation`: Aggregate execution metrics (e.g., scenario count based on step count).
- `artifacts`: Link to primary and supporting logs.

### 2. Archive Generation
Implement a zipping mechanism:
- Gathers the raw trace JSONs, diffs, and the manifest.
- Computes SHA256 checksums.
- Zips into `TP-<ID>_PROOF_ARCHIVE.zip`.
- Updates the final `_PROOF_BUNDLE.json` artifact block with the archive's metadata.

### 3. CLI Integration
- Add a new CLI command or extend `tp exec` to trigger the final aggregation step automatically upon successful execution.

## Verification & Testing
- Execute a sample TP and verify that the resulting `_PROOF_BUNDLE.json` passes strict JSON schema validation against `proof/standards/PROOF_BUNDLE_SCHEMA.json`.
- Verify the generated `.zip` file contains the correct manifest and unzips properly.

## Migration & Rollback
Additive changes to the observability module.
